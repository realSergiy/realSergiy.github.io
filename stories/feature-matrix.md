A source-code-derived comparison of seven coding agents found in this directory. Every agent was reviewed against an identical 58-feature taxonomy and scored **0–100** based **only on source-code evidence** (not marketing/README claims). Scores: `0` = absent, `1–25` = stub/partial, `26–50` = basic, `51–75` = solid, `76–90` = comprehensive, `91–100` = best-in-class.

## Agents under review

| Key | Agent | Language / Stack | One-line identity |
|-----|-------|------------------|-------------------|
| aider | **aider** | Python (LiteLLM) | Terminal pair-programmer; repo-map + SEARCH/REPLACE + deep git |
| claude-code | **claude-code** | TypeScript / Bun (Ink TUI) | Anthropic-only agent with best-in-class hooks/skills/plugins/safety |
| hermes | **hermes-agent** | Python | Huge multi-channel assistant; LSP, cron, skills, browser/computer-use |
| kilocode | **kilocode** | TypeScript / Bun (Effect-TS, OpenCode fork) | CLI + VS Code/JetBrains/Zed; vector RAG, worktree Agent Manager |
| openclaw | **openclaw** | JS/TS (gateway + 131 plugins) | Plugin-centric multi-channel platform; voice, media-gen, desktop apps |
| opencode | **opencode** | Go (Bubble Tea TUI) | Clean Go terminal agent; LSP diagnostics, permissions, SQLite |
| pi | **pi** | TypeScript monorepo | Minimalist-by-design; rich provider/streaming layer + extension ecosystem |

## Feature Descriptions

Definitions used to score each feature (these explain what each row name means).

### A. Models & Providers
1. **Multi-provider LLM support** — Breadth of supported model vendors/APIs (OpenAI, Anthropic, Google, OpenRouter, Bedrock, Azure, etc.) and how cleanly providers are abstracted.
2. **Local / self-hosted models** — First-class support for running against local/offline inference servers (Ollama, llama.cpp, LM Studio, vLLM, SGLang) beyond a generic OpenAI-compatible base URL.
3. **Model routing / per-role models** — Ability to assign different models to different roles/tasks (e.g. a planner vs. an editor vs. a cheap summarizer model) and to switch models at runtime.
4. **Prompt caching** — Explicit use of provider prompt-cache mechanisms (cache_control / ephemeral markers, TTLs, cache-key stability) and accounting of cache-read/write tokens.
5. **Reasoning / extended-thinking** — Support for reasoning/thinking models (thinking budgets, reasoning-effort levels, think-tag handling, interleaved thinking, signature replay).
6. **Streaming responses** — Token/event streaming from the model to the UI with granular event types and fallbacks.

### B. Code Editing
7. **Search/replace (diff) editing** — A SEARCH/REPLACE or old_string→new_string block edit format for targeted changes.
8. **Whole-file editing** — Creating or fully overwriting a file's contents.
9. **Unified-diff editing** — Applying edits expressed as a unified/`@@`-hunk diff (or an equivalent structured patch envelope) as the *input* edit format.
10. **Multi-file coordinated edits** — Applying changes across multiple files within one operation/turn, ideally atomically.
11. **Edit validation & auto-repair** — Detecting failed/ambiguous edits or post-edit syntax/lint/type errors and feeding them back so the model self-corrects.
12. **Fuzzy / flexible match application** — Tolerant application of edits when the model's text doesn't match byte-for-byte (whitespace/indentation/Unicode normalization, similarity matching).

### C. Codebase Understanding
13. **Repo map / codebase indexing** — A precomputed map or ranked context-selection mechanism that summarizes the repository's structure/symbols for the model.
14. **Tree-sitter / AST parsing** — In-process syntactic/AST parsing of source code (for symbol extraction, chunking, navigation, or security analysis).
15. **Semantic / embedding search (RAG)** — A vector/embedding index enabling semantic retrieval over the codebase (or memory).
16. **Grep / regex / glob search** — Lexical search tools (ripgrep/grep, glob, fd) exposed to the agent.
17. **LSP integration** — A language-server client providing diagnostics, hover, go-to-definition, references, symbols.
18. **Shell / command execution** — Running shell/terminal commands, ideally with persistent sessions, timeouts, and background processes.

### D. Tools & Capabilities
19. **Tool / function-calling framework** — A general tool abstraction/registry (schemas, dispatch, hooks, filtering) rather than hardcoded behaviors.
20. **MCP client** — Consuming external Model Context Protocol servers (stdio/SSE/HTTP, OAuth, resources/prompts).
21. **MCP server** — Exposing the agent itself (or built-in servers) over MCP for other clients.
22. **Web search** — Querying a web search engine/provider.
23. **Web / URL fetch & scrape** — Fetching URLs and converting page content to text/markdown (with safety guards).
24. **Browser automation** — Driving a real/headless browser (CDP/Playwright) for clicking, typing, navigation, screenshots.
25. **Image / multimodal input** — Accepting images/PDFs as model input (vision), with format handling and model gating.
26. **Image / media generation** — Generating images/video/audio via generative models.

### E. Agent Loop & Orchestration
27. **Autonomous agentic loop** — A multi-step reason→act→observe loop that executes tools and iterates toward a goal.
28. **Planning / plan mode** — A dedicated planning/architect mode (often read-only) that produces a plan before execution.
29. **Sub-agents / task delegation** — Spawning child agents to handle sub-tasks (with isolation, restricted toolsets, cost roll-up, fan-out).
30. **Todo / task-list tracking** — A structured to-do/task-state mechanism the agent maintains across a session.
31. **Custom agent modes / personas** — User-definable agents/modes/personas with their own prompts, tools, or models.
32. **Scheduled / background / cron tasks** — Running agent work on a schedule or in the background asynchronously.

### F. Context & Session
33. **Context compaction / token management** — Summarizing/truncating conversation to stay within token budgets (auto-compact thresholds, overflow handling).
34. **Conversation persistence & history** — Durable storage of conversation/messages/tool calls (files or a database).
35. **Session resume / restore** — Resuming/continuing prior sessions (`--continue`/`--resume`, pickers, forking).
36. **Cost / token usage tracking** — Tracking and reporting token usage and dollar cost (including cached-token accounting).
37. **Long-term / cross-session memory** — Persistent memory that carries learned facts/context across sessions (beyond static rule files).

### G. Git & Review
38. **Git auto-commit / integration** — Built-in git operations, especially automatic commits of agent edits with generated messages and authorship handling.
39. **Undo / change review / diff** — Presenting diffs and reverting changes (checkpoints/snapshots/undo).
40. **PR / code-review automation** — Creating pull requests and/or automating code review of diffs/PRs.

### H. Interfaces
41. **CLI / headless mode** — Non-interactive/scriptable invocation (single-shot prompts, JSON/stream output, batch runs).
42. **Interactive TUI** — A rich interactive terminal UI (streaming render, dialogs, autocomplete, themes).

### I. Extensibility

## Notes & Caveats

- **Scoring is comparative and source-based.** A `0` means no source evidence was found; it does not always mean the capability is impossible (e.g. local models often work via a generic OpenAI-compatible path even where a first-class integration is absent).
- **Different products, different goals.** `claude-code` is deliberately single-vendor (Anthropic) but extremely deep on safety/extensibility; `pi` is deliberately minimalist (no MCP/sub-agents/plan-mode/permissions by design, pushed to extensions); `openclaw`/`hermes` are broad multi-channel *platforms* whose "coding agent" is one facet; `aider` and `opencode` are focused terminal coding tools. Averages at the bottom of Table 1 are a rough aggregate only and weight all 58 features equally.
- **`kilocode` identity:** the reviewed code is the **Kilo CLI**, an Effect-TS fork of **OpenCode** (with VS Code/JetBrains/Zed front-ends), *not* the Roo-Code/Cline-lineage extension.
- **`claude-code` identity:** this is a full TypeScript/Bun implementation of the Claude Code CLI (custom Ink TUI, `query.ts`/`QueryEngine.ts` loop).
