# Upstream Technical Analysis: Fedora Linux 44 and Arch Linux in May 2026

## 1. Release Model and Lifecycle

The foundational divergence between Fedora Linux and Arch Linux originates in their respective release engineering models, which subsequently dictate system architecture, software currency, and the end-user administration lifecycle. These contrasting methodologies define how code transitions from upstream project repositories to local disk execution.

Fedora Linux operates on a strict, time-based point-release model, delivering a new primary OS version biannually. As of May 2026, the current iteration is Fedora 44, which was finalized and released on April 28, 2026. This cadence is systematically engineered to construct a predictable, static target for enterprise derivatives and third-party software developers. Each release effectively serves as a frozen Application Binary Interface (ABI) and Application Programming Interface (API) baseline. Core system libraries (such as `glibc`, `openssl`, and fundamental compiler toolchains) maintain their major versions throughout the 13-month support lifecycle of the release. Upgrades between major versions are orchestrated as discrete, transactional state mutations. Utilizing the `dnf system-upgrade` mechanism—now natively executed by the DNF5 package manager—Fedora downloads the complete delta of the new release, reboots into a specialized target, and applies the package transactions offline during the boot sequence. This offline methodology structurally prevents the dependency resolution collisions and live-memory library corruptions that can occur when replacing base system primitives on a running desktop environment.

To sustain this point-release architecture, Fedora maintains a highly formalized pre-release staging pipeline. Code initially enters "Rawhide," the continuous, rolling development branch where integration testing occurs. Prior to a formal release like Fedora 44, the codebase is structurally branched off from Rawhide, moving through strictly gated Alpha and Beta phases. These phases are governed by rigid "blocker-bug" criteria—specific functional requirements (e.g., installer bootability, display manager initialization, core network resolution) that must pass automated and manual openQA testing before the release is declared golden.

Conversely, Arch Linux employs a pure rolling-release architecture. The concept of major release versions, ABI freezes, and discrete offline upgrade paths is entirely absent from the upstream project design. The "May 2026" iteration of Arch Linux is not a distinct product version, but rather a temporal snapshot represented by the 2026.05.01 ISO. The upgrade mechanism is a continuous, live-system process executed directly on the running filesystem via the `pacman -Syu` command. Arch does not segregate core system component upgrades from standard application updates; a routine daily synchronization may seamlessly and concurrently transition the system from GNOME 49 to GNOME 50, or from an older compiler to a new one, all while the user operates the system.

Because Arch lacks frozen release branches, it inherently does not provide Long Term Support (LTS) system states. The entire operating system rolls forward synchronously. However, Arch does recognize the necessity of hardware interface stability and curates specific LTS components to provide a localized buffer against upstream volatility. Most notably, the `linux-lts` package (shipping Linux 6.18.25 in the May 2026 window alongside the mainline 7.0.3 kernel) allows administrators to pin critical hardware abstraction layers while the userland continues its continuous forward trajectory. Arch’s pre-release testing infrastructure is vastly more constrained than Fedora's. It relies on the `core-testing` and `extra-testing` repositories. These repositories function as brief, parallel staging grounds primarily used for testing major toolchain updates or detecting catastrophic build failures. They do not undergo the prolonged, formalized QA branching seen in Fedora; instead, they rely on rapid integration and immediate feedback from a subset of advanced users prior to migrating the packages to the main repositories.

## 2. Package Management and Repositories

The core technical mechanisms by which software dependencies are resolved, payloads downloaded, and binaries integrated into the local filesystem define the day-to-day operational reality for administrators.

Fedora 44 represents the culmination of a multi-year architectural transition, officially solidifying DNF5 as the default package manager across all new installations and upgrades, fully obsoleting DNF4. Written entirely in C++, DNF5 eliminates the historical Python interpreter dependency and the heavyweight metadata overhead that plagued its predecessor. This architectural rewrite yields drastic reductions in memory consumption and execution latency, particularly on low-resource environments. DNF5 operates on the standard RPM binary format and utilizes `libsolv`—a highly sophisticated, strict boolean SAT solver—to resolve incredibly complex dependency trees and modular overlapping. This advanced dependency resolution supports highly parallelized metadata downloading and fully transactional operations. Every package state change is recorded, allowing for precise rollbacks or environment undo operations via `dnf history` directives.

Arch Linux relies on Pacman, which reached version 7.1.0 in the May 2026 window. Written in pure C, Pacman prioritizes execution velocity and architectural simplicity over complex solver heuristics. Instead of a heavyweight SAT solver, it uses a simpler directed graph approach to resolve dependencies. Packages are distributed as standard `bsdtar` archives compressed with the `zstd` algorithm (`.pkg.tar.zst`). The binary package format is effectively a compressed filesystem tree coupled with straightforward shell-script metadata. This simplicity makes Pacman exceptionally fast, but it places the burden of correct dependency mapping and conflict resolution entirely on the package maintainer and the end user, as the tool cannot gracefully negotiate complex, multi-path version conflicts automatically. A critical technical security advancement in the Pacman 7.x series is the implementation of the `DownloadUser` privilege separation mechanism. Pacman now automatically drops root privileges during the payload fetching phase, mapping the network operation to a separate unprivileged user, thereby significantly mitigating surface-area attacks on the network stack during database synchronizations.

Repository structure highlights different philosophies regarding software curation. Fedora strictly divides its software into the core `fedora` repository (the baseline packages at the time of the major release) and the `updates` repository (patches and minor version bumps added over the 13-month lifecycle). Third-party software curation is handled via COPR (Community Projects). COPR is fundamentally a server-side build system: contributors submit source code and RPM spec files, and Fedora's infrastructure (utilizing Mock and Koji) securely compiles the software into RPM binaries within isolated environments. The end user simply adds the COPR repository and downloads pre-compiled, cryptographically signed binaries.

Arch Linux consolidates its official packages into `core` (essential system components, kernels, toolchains) and `extra` (all other software, having recently absorbed the legacy `community` repository). For third-party software, Arch provides the Arch User Repository (AUR). Unlike Fedora's COPR, the AUR does not host any pre-compiled binaries; it is purely a Git repository of `PKGBUILD` text scripts. The end-user workflow involves utilizing `makepkg` to download the `PKGBUILD`, pull upstream sources directly from the developer (e.g., GitHub, GitLab), apply local patches, and compile the software locally on the user's hardware. This necessitates deep user auditing, as `PKGBUILD` scripts execute arbitrary bash commands on the local machine during the build phase.

At the frontier of system deployment, Fedora provides fully supported, atomic, immutable desktop variants at the upstream level (Fedora Silverblue, Kinoite, Sway Atomic, Cosmic Atomic). In Fedora 44, these deployments are aggressively transitioning toward "bootable containers" (`bootc`) utilizing `composefs` and `fs-verity` for cryptographic read-only filesystem verification, integrating native OSTree delta updates, and defaulting to Flatpak for all graphical user-space applications. Arch Linux does not maintain any upstream atomic or immutable variants, retaining a strictly mutable root filesystem governed purely by pacman state changes, and includes no universal packaging (Flatpak/Snap) by default, requiring explicit user configuration.

Parallel versioning is handled elegantly in Fedora via explicitly versioned package nomenclature (e.g., shipping both `mariadb-10.11` and `mariadb-11.8` simultaneously) combined with the `alternatives` system. Arch Linux actively discourages parallel versions of software by design. Arch forces system-wide transitions to the newest library standard immediately, relying on occasional AUR packages (e.g., `gcc14`, `gcc13`) for legacy compilation targets rather than native repository support.

## 3. Software Currency and Kernel Policy

The velocity of software delivery is the most tangible manifestation of the underlying release models, directly impacting developer access to modern APIs and end-user access to hardware enablement.

As of May 2026, Fedora 44 ships with a heavily modernized stack systematically synchronized to its April release date. The distribution provides GNOME 50, KDE Plasma 6.6 (featuring the new unified Plasma Login Manager replacing SDDM by default), LLVM 22, GCC 16.1, Glibc 2.43, and the Linux 7.0.x kernel series (patched to 7.0.9 during the late May window). Arch Linux’s 2026.05.01 ISO and subsequent rolling updates provide nearly identical userland versions in real-time, including GNOME 50.1, KDE 6.6.4, Glibc 2.43, and Linux 7.0.3.

However, a critical divergence exists in massive core compiler toolchains. Because Fedora explicitly coordinates mass package rebuilds prior to its spring release, it was able to aggressively integrate GCC 16.1 across the entire distribution architecture, forcing all packages to compile against the new baseline. Arch Linux, prioritizing continuous rolling stability over synchronized mass rebuilds, is operating on GCC 15.2.0 as of May 2026. Arch will eventually transition to GCC 16, but this dynamic demonstrates that Fedora’s point-release model can occasionally leapfrog pure rolling releases on core structural toolchains when upstream release timelines neatly align with Fedora's branching schedule.

Regarding kernel policy, Arch Linux ships an almost entirely vanilla kernel directly from `kernel.org`. The downstream patching is microscopic, consisting almost exclusively of minor arch-specific build fixes to satisfy Pacman packaging requirements. Out-of-tree modules are not integrated into the base kernel package, requiring DKMS (Dynamic Kernel Module Support) for integration. Fedora, while closely tracking mainline, applies a noticeably thicker distribution patch set to its kernel. This is done to explicitly enable features deemed necessary for the Workstation experience, or to backport critical hardware enablement prior to upstream consolidation.

A prominent technical example of this policy divergence in the May 2026 era is the handling of the `NTSYNC` kernel module. NTSYNC, integrated into the mainline kernel starting in 6.14, provides a native, in-kernel implementation of Windows NT synchronization primitives, drastically reducing overhead for Wine and Proton and eliminating the need for older Fsync/Esync user-space workarounds. Fedora 44 aggressively configures and enables this module by default via package recommendations; installing Wine or Steam triggers the `wine-ntsync` package, instantly optimizing gaming performance with zero user intervention. Arch Linux users running the 7.0.x kernel also possess the NTSYNC module, but because of Arch's strict upstream adherence and lack of "magic" configuration scripts, users must manually load the module (e.g., `echo ntsync > /etc/modules-load.d/ntsync.conf`) and manually configure their Wine/Proton environment variables to reap the identical benefits.

## 4. Installation and First-Boot

The initial provisioning phase underscores the target demographic and administrative assumptions of each distribution. Fedora 44 utilizes Anaconda, a highly complex, graphical, and scripted installer deeply tied to Red Hat’s Kickstart automated provisioning ecosystem. Anaconda is designed to abstract away the deep complexities of disk partitioning, bootloader installation, and network configuration, offering a rapid path to a working system. In Fedora 44, a notable technical modification restricts Anaconda from creating default network profiles for all detected hardware devices. Instead, it limits configuration solely to network interfaces actively utilized during the installation phase, a change engineered to prevent post-install state contamination and networking conflicts. The first-boot experience on Fedora Workstation invokes a graphical GNOME Initial Setup wizard, guiding the user seamlessly through user creation, Wi-Fi setup, and explicit opt-in prompts for third-party repositories and telemetry data collection.

Arch Linux lacks a traditional graphical installer entirely. The May 2026 ISO boots directly into a live Zsh root shell. The classical installation method is entirely manual: the administrator executes raw partitioning tools (`fdisk`, `parted`), formats filesystems via `mkfs`, mounts the topology, and utilizes the `pacstrap` script to bootstrap the base system binaries. To lower the barrier to entry, Arch provides `archinstall` (version 4.3 in the May 2026 ISO), a Python-based TUI (Terminal User Interface) scripted installer. `Archinstall` automates standard boilerplate configurations—such as installing Wayfire Wayland compositors, adding complex Unicode font matrices (Noto CJK), and orchestrating Btrfs filesystem setups—but still requires the user to possess a fundamental understanding of Linux storage topologies. Furthermore, there is no centralized first-boot wizard. Upon reboot, the system state is exactly what the user commanded via the CLI or `archinstall`; it usually drops directly to a TTY login prompt unless a display manager (like SDDM or GDM) was explicitly selected and enabled.

Fedora produces highly curated "Editions" (Workstation, Server, IoT, CoreOS) and "Spins" (KDE, Sway, Cosmic, Budgie), each acting as a distinct product with tailored default packages and customized configurations. Arch Linux produces a single, universal base ISO. The concept of an "edition" does not exist at the upstream level, as every Arch system is bespoke, built modularly by the user.

## 5. System Architecture and Defaults

While both systems natively utilize `systemd` as the init system and service manager, their architectural integration philosophies contrast sharply, dictating how an administrator interacts with the daemon lifecycle.

Fedora patches and pre-configures `systemd` heavily to ensure immediate utility. It utilizes `systemd-presets` to aggressively enable services upon package installation. In practical terms, if a user installs a web server like Apache or a virtualization daemon like `libvirtd` on Fedora, the service is automatically enabled and often started instantly. Arch Linux adheres to a strict "no-surprises" ideology. Installing a package simply places the software binaries onto the disk. The service remains completely inert until the user explicitly issues a `systemctl enable --now` command. This prevents unintended background processes from executing and opening network ports merely because a package was resolved as a dependency.

Filesystem defaults represent another massive operational divergence. Fedora 44 defaults to the Btrfs filesystem for Workstation installations. However, its default subvolume layout is relatively flat, typically separating only the root (`/`) and home (`/home`) directories. Crucially, Fedora does not automatically integrate Btrfs snapshots into the bootloader or package manager for seamless rollback capabilities; achieving a Snapper-compatible layout requires manual intervention during or after Anaconda installation. In contrast, while manual Arch installations leave filesystems entirely to user discretion, the `archinstall` 4.3 script natively supports configuring Btrfs with a complex, hierarchical subvolume layout (e.g., `@`, `@.snapshots`) perfectly optimized for Snapper and `grub-btrfs` integrations. Ironically, an automated Arch install in May 2026 provides a more robust snapshot rollback foundation out-of-the-box than a default Fedora Workstation installation.

Bootloader topology in Fedora revolves heavily around GRUB2, utilizing `dracut` to generate the initramfs. While Fedora is laying the active groundwork for Unified Kernel Images (UKI) and `systemd-boot`—visible in its Phase 1 bootloader update unifications and the new sealed atomic container images —the standard mutable Fedora 44 still defaults to GRUB for broad legacy compatibility. Arch relies on `mkinitcpio`, an Arch-specific bash-driven framework, for initramfs generation. Arch officially supports GRUB, `systemd-boot`, EFISTUB, and rEFInd equally, though `systemd-boot` has rapidly become the de facto standard recommended by modern Arch documentation for UEFI systems due to its simplicity and native UKI integration capabilities.

Regarding audio stacks, both distributions utilize PipeWire. However, Fedora configures PipeWire alongside the WirePlumber session manager natively out of the box, ensuring complex Bluetooth and PulseAudio-bridge routing functions immediately. Arch packages these exact upstream components but requires the user to explicitly install `pipewire-pulse` and `wireplumber` to achieve identical desktop audio functionality.

Finally, the configuration philosophy of `/etc` showcases differing views on upstream trust. Fedora heavily pre-populates `/etc` with opinionated, distribution-specific defaults designed to unify the user experience. Arch Linux treats upstream configuration files as sacrosanct; `/etc` is mostly bare, and when configuration files are provided, they reflect exactly what the original software author intended, unmodified by Arch maintainers.

## 6. Hardware and Firmware

The policies governing hardware enablement highlight differing interpretations of free software mandates. Fedora targets Tier-1 enterprise and workstation hardware, officially supporting the `x86_64` and `aarch64` architectures at the upstream level. It maintains a strict ideological stance on free software for its core repositories. While non-free firmware binaries (like WiFi card blobs) are allowed to ensure the hardware can physically boot, proprietary drivers (most notably NVIDIA graphics drivers) and encumbered media codecs are explicitly excluded from the base OS. Users must enable third-party RPM Fusion repositories to attain proprietary hardware acceleration and commercial codec playback.

Arch Linux officially supports only the `x86_64` architecture (an independent, downstream community project handles ARM implementations). However, Arch takes a strictly pragmatic stance on hardware enablement. The official `extra` repository contains proprietary NVIDIA drivers (`nvidia`, `nvidia-dkms`), allowing users to seamlessly install, update, and manage closed-source hardware blobs directly through `pacman` without configuring precarious third-party repositories. Both distributions aggressively track `linux-firmware` and CPU microcode packages, ensuring day-one support for new peripheral and component architectures.

## 7. Security and Supply Chain

Security posture reveals disparate approaches to threat modeling and administrative overhead. Fedora 44 defaults to SELinux (Security-Enhanced Linux) in enforcing mode, applying a stringent, pre-configured Mandatory Access Control (MAC) policy across the entire OS. Furthermore, Fedora works out-of-the-box with Microsoft-signed Secure Boot architectures via the signed `shim` bootloader, requiring zero manual key enrollment by the user to boot securely. The Fedora project maintains a highly formal CVE response infrastructure, issuing FEDORA-* errata rapidly to patch vulnerabilities without breaking the surrounding ABI.

Arch Linux ships without any Mandatory Access Control frameworks enabled by default (AppArmor or SELinux can be installed manually but lack deep, integrated default profiles). Secure Boot on Arch requires advanced manual intervention. Because Arch does not possess a Microsoft-signed shim, users must use tools like `sbctl` to generate their own custom Secure Boot keys, physically place their motherboard firmware into Setup Mode, enroll the custom keys into the TPM/NVRAM, and generate Unified Kernel Images (UKIs) signed automatically with pacman hooks. Arch tracks vulnerabilities via Arch Security Advisories (ASA), delivering patches rapidly but placing the burden of applying them—and dealing with any subsequent upstream version changes—entirely on the user.

In terms of supply chain integrity, the distributions flip advantages. Fedora isolates all builds inside the Koji build system, ensuring a clean environment and cryptographic package signing, but it does not mandate bit-for-bit reproducible builds. Arch Linux, conversely, has made massive, industry-leading strides in deterministic compilation. By May 2026, Arch achieves approximately 98% reproducible builds across its entire repository. By rigorously controlling `SOURCE_DATE_EPOCH` timestamps, eliminating arbitrary code execution in build hooks, and recording `BUILDINFO` metadata, independent auditors can compile the exact same binary hash from upstream source. This mathematically verifies that the Arch build servers were not compromised during compilation.

## 8. Performance and Footprint

The installation footprint reflects the distributions' target demographics. Fedora 44 Workstation carries a heavier structural footprint, requiring nearly 20GB of disk space for a standard installation and hovering around 1.5GB to 2GB of idle RAM consumption depending on the desktop environment. Arch Linux’s base footprint is minimal; a foundational TTY installation consumes under 2GB of disk space and roughly 150MB of RAM, scaling up dynamically only based on the exact desktop components and services the user explicitly adds.

Compiler build tuning defines runtime execution efficiency. Both distributions nominally target standard `x86_64` as a baseline, but Fedora has aggressively integrated advanced microarchitecture targeting. Through the `glibc-hwcaps` mechanism, Fedora 44 seamlessly supports `x86-64-v2` (adding POPCNT, SSE4.2) and `x86-64-v3` (AVX, AVX2) instructions. Fedora achieves this by placing optimized variants of binaries in separate directories, utilizing a symlink wrapper in `/usr/bin` to query the host CPU at runtime and execute the most highly optimized variant available without crashing older hardware.

Arch Linux avoids the storage and compilation overhead of multi-architecture packages, sticking to a single `x86_64` binary target for its official repositories. However, to maximize performance universally, Arch enables Link Time Optimization (`-flto`) by default globally within its `makepkg.conf` macro definitions. LTO allows the GCC compiler to analyze and optimize code execution paths across the entire program during the linking phase, resulting in smaller, faster executables that offset the lack of higher explicit microarchitecture targeting.

## 9. Stability and Update Workflow

Fedora’s 6-month cycle is governed by stringent Quality Assurance (QA). Blocker bugs regarding bootability, display rendering, and package manager integrity must be resolved before a release goes golden. Automated testing via openQA ensures that updates deployed to the `updates` repository rarely brick a system. Rollbacks, if necessary, are executed effectively via DNF5's transactional history commands, or handled natively and instantly via OSTree rebases on the Atomic Desktop variants.

Arch Linux operates entirely on user responsibility and upstream trust. Because it rolling-releases complex interdependencies constantly, the Arch QA process relies on rapid testing in the testing repos and immediate community feedback. A fundamental, technically enforced policy of Arch is that **partial upgrades are strictly unsupported**. Running `pacman -Sy` (updating metadata) without `-Su` (upgrading packages), and subsequently installing a new package, can link a newly compiled binary against an outdated system library, destroying the local filesystem state. Arch users are explicitly expected to check the Arch Linux front-page news feed before executing updates, as structural changes (e.g., the 2026 migration to `iptables-nft` or system user changes for DHCP daemons) require manual configuration file (`.pacsave`/`.pacnew`) interventions that pacman will not resolve automatically.

## 10. Workload Suitability

Evaluating these distributions against specific computational workloads highlights their structural philosophies. For a daily-driver desktop, Fedora Workstation provides an incredibly polished, frictionless GNOME or KDE experience. For enterprise server deployments, Fedora Server directly feeds into RHEL, utilizing identical methodologies. Arch Linux is rarely deployed in critical enterprise server environments due to the lack of compliance certifications and its rolling ABI, making automated server fleet maintenance highly unpredictable.

For modern Artificial Intelligence and Machine Learning (AI/ML) workflows, the dynamic shifts dramatically in favor of Arch. Packaging the massive, proprietary software stacks required for GPU computation (NVIDIA CUDA, AMD ROCm) conflicts deeply with Fedora’s strict FOSS licensing. Fedora users must rely on external containerized workflows, complex RPM Fusion setups, or manual binary installations. Arch Linux, unrestrained by such licensing mandates, packages these monoliths directly into the official `extra` repository. In May 2026, an Arch user can deploy a fully functional, bare-metal AI development environment in minutes by simply executing `pacman -S cuda cudnn python-pytorch-cuda rocm-hip-sdk`.

For Gaming, both distributions excel due to rapid graphics stack updates (shipping Mesa 26.x in May 2026). The integration of the NTSYNC kernel module in the Linux 7.0 era massively boosts Windows game compatibility and frame pacing via Wine and Proton. As noted, Fedora ships it enabled by default; Arch requires a trivial manual configuration to achieve the identical, massive performance uplift.

In Embedded and Edge computing scenarios, Fedora dominates. Fedora IoT and CoreOS are specifically built for edge immutability, leveraging `rpm-ostree` and `bootc` for secure, atomic fleet updates. Arch Linux produces minimal images, but rolling an embedded edge device is notoriously risky without immense manual rollback engineering.

From an educational standpoint, Arch Linux offers exceptionally high value. Building an Arch system manually forces the user to understand bootloaders, filesystem tables (`fstab`), and service managers—knowledge that Fedora’s Anaconda installer intentionally abstracts away.

## 11. Documentation and Community

The ArchWiki is universally recognized as the premier documentation source across the entire Linux ecosystem, frequently utilized by administrators of entirely different distributions. It is structured as an exhaustive, encyclopedic reference manual. It details not just how to install a package on Arch, but the theoretical underpinnings, configuration syntax, and internal mechanics of the software itself.

Fedora Docs adopts a distinctly product-oriented approach. The documentation is structured as modular quick-starts and administrative guides specifically focused on how to execute tasks using Fedora’s specific tooling (e.g., Cockpit, Firewalld, SELinux, DNF5). It lacks the universal breadth of the ArchWiki but provides highly focused, curated, and enterprise-relevant instructions for deploying Red Hat technologies.

Regarding community channels, Arch relies on highly moderated Forums, IRC, and GitLab issue trackers. The Arch community culture expects users to have exhaustively read the manual before asking questions. Fedora utilizes Fedora Discussion (Discourse), Ask Fedora, and a robust Bugzilla tracker, generally fostering a more tolerant environment for beginner inquiries.

## 12. Licensing, Telemetry, and Freedom Posture

Ideological posture manifests directly in technical implementation. Neither distribution is endorsed by the Free Software Foundation (FSF) because both permit the inclusion of non-free software, but their administrative boundaries differ sharply. Fedora enforces a strict FOSS mandate for its official repositories; non-free firmware is permitted (to ensure hardware can physically initialize), but non-free userland applications, proprietary drivers, and commercial codecs are strictly banished. Arch Linux is completely pragmatic; if a proprietary software package is highly utilized by the community (e.g., Discord, NVIDIA, Steam, CUDA), it is packaged directly in the official `extra` repository for administrative convenience.

Telemetry data collection has become a significant technical differentiator. In Fedora 44 (building upon the implementation approved for F42), Fedora Workstation includes a privacy-preserving, opt-in telemetry system based on the Endless OS Foundation metrics framework. During the first-boot GNOME Initial Setup phase, users are presented with a toggle to upload generic hardware profiles and desktop usage patterns. Crucially, this toggle has no default value, forcing the user to make an explicit choice before proceeding. The data undergoes rigorous client-side filtering and anti-fingerprinting (metrics are intentionally decoupled so they cannot be linked together to form a unique system profile) before transmission. Users who upgrade from previous versions rather than performing clean installs have telemetry disabled by default.

Arch Linux handles data collection via `pkgstats`, an entirely separate, passive package that the user must manually discover, install, and enable via a systemd timer. It securely sends a simple, anonymous list of installed packages and basic CPU architecture data to Arch developers to inform package maintenance priorities.
