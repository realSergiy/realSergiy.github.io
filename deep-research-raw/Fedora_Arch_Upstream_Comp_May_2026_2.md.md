# Fedora 44 vs Arch Linux (May 2026): Upstream-to-Upstream Technical Comparison

**Scope:** Fedora Linux 44 (released 2026‑04‑28) versus a freshly updated Arch Linux rolling system whose May 2026 snapshot is represented by the 2026.05.01 ISO (kernel 7.0.3). Downstreams (RHEL/CentOS Stream/Alma/Rocky on the Fedora side; CachyOS/Manjaro/EndeavourOS/SteamOS/Garuda/Artix on the Arch side) are excluded. Identical points (systemd, Wayland by default, Bash, same DE list) are skipped.

## TL;DR
- Fedora 44 ships a fixed, QA-gated point release with kernel 6.19.14-300.fc44, GCC 16.1, GNOME 50, SELinux enforcing, dnf5/RPM 6.0, Btrfs+snapm by default, and a stated ≥99% reproducible-build target; Arch in May 2026 is a vanilla rolling system with kernel 7.0.3, GCC 16.1.1, GNOME 50.1, no MAC by default, no default filesystem, and pacman 7.1.
- Net consequence: Fedora is faster to adopt coordinated new desktop/toolchain rebases under QA gates, while Arch is faster to adopt new kernels and individual upstream releases by skipping curation; Arch users in May 2026 are ahead on kernel (7.0.3 vs 6.19.14) and individual packages by days-to-weeks, while Fedora is ahead in providing rollback, MAC, FIPS, Secure Boot, immutable variants, and multi-arch support as defaults.
- Pick Fedora if you want SELinux/Secure Boot/snapshots/crypto-policies/immutable variants/non-x86_64 from upstream; pick Arch if you want minimal opinionation, AUR access (112,669 PKGBUILDs as of May 2026), and a system whose every default you set yourself.

## Key Findings
- Both ship the same major upstream stack (GNOME 50.x, Plasma 6.6.x, Mesa 26.0.x, PipeWire 1.6.x, glibc 2.43, GCC 16.1), so user-visible *components* are equivalent in May 2026; the difference is everything around them.
- Fedora's curation surface is large: Anaconda installer, Btrfs+snapm/boom snapshots, SELinux targeted policy, dracut UKIs for Atomic, Koji+Bodhi QA, FEDORA-* errata, countme telemetry on by default, dnf system-upgrade, RPM Fusion as third-party codec channel.
- Arch's curation surface is deliberately small: pacstrap → user choice for FS, bootloader, initramfs; mkinitcpio default; AUR with PKGBUILD scripts and no vetting; ASAs only for core/extra/multilib; pkgstats is opt-in only.
- Arch is x86_64-only (v1 baseline); Fedora supports x86_64 (v1 baseline + glibc-hwcaps), aarch64, ppc64le, and s390x, with the Asahi Remix for Apple Silicon.

## Details

### 1. Release model and lifecycle

| Criterion | Fedora | Arch | Net difference / notes |
|---|---|---|---|
| **1. Release model & lifecycle** | Point release every ~6 months (F44 released 2026‑04‑28 after two delays from the original 14 Apr target). Each release supported ~13 months (current + previous; F42 EOL'd at F44 GA). Upgrade via `dnf system-upgrade --releasever=N`. No LTS. | Pure rolling. ISO snapshots are monthly *install media only* (2026.05.01 = kernel 7.0.3, 1.4 GB); existing systems update via `pacman -Syu`. No versions, no upgrade path, no LTS. | Fedora trades freshness within a release for an integration point; Arch trades stability windows for continuous delivery. The Arch ISO is a convenience artifact — there is no concept of "Arch 2026.05" as a release. |
| **2. Testing/pre-release branches** | Rawhide (continuously rebuilt rolling dev branch), Branched (frozen N+1 in stabilization, e.g. f45-branched mid-2026), `updates-testing` (per-release staging), Bodhi karma gates promotion. | `core-testing`, `extra-testing`, `multilib-testing`; `*-staging` for mass rebuilds (devs only, explicitly "will break"); `gnome-unstable`, `kde-unstable` for DE pre-releases. No equivalent of Rawhide-as-distro (closest is testing repos). | Fedora ships its dev branch (Rawhide) as a usable distro; Arch's `*-testing` is opt-in by editing `/etc/pacman.conf`. Arch has no public daily-snapshot equivalent of Rawhide. |

### 2. Package management and repositories

| Criterion | Fedora | Arch | Net difference / notes |
|---|---|---|---|
| **3. Package manager & format** | `dnf5` (libdnf5; F44 also switches PackageKit to the DNF5 backend); RPM 6.0; transactions with full history/rollback (`dnf history undo/rollback`), parallel downloads, weak deps (`Recommends:`/`Suggests:`), versioned RPM conflict resolution. | `pacman 7.1` (Nov 2025); `.pkg.tar.zst` (zstd-compressed); transactional install set but no history-based rollback (downgrade requires Arch Linux Archive or cached `.pkg.tar.zst`); parallel downloads (since 6.0); sandboxed downloads via `DownloadUser`, Landlock + `NO_NEW_PRIVS`. | Fedora's resolver/transaction system is more featureful (history, undo, weak deps). Arch's pacman is faster and simpler; **no real rollback** without external mechanisms (snapshots, ALA, paccache). For day-to-day work pacman feels snappier; for recovery, dnf is more forgiving. |
| **4. Repository structure** | `fedora` (release frozen), `updates`, `updates-testing`, `fedora-cisco-openh264` (Cisco-licensed H.264). All packages signed since 2012 (mandatory in pacman 4+). ~14,000 binary packages. | Fedora ships ~6× the binary packages and treats releases as immutable trees; Arch treats every repo as a moving target. Both rely on per-package signatures with the package DB itself signed. |
| **5. Third-party & user-contributed packages** | **COPR** (build-as-a-service in Fedora infrastructure; each user namespace; per-user signing key; equivalent to PPAs). **RPM Fusion** is the canonical channel for non-free/patent-encumbered content (multimedia codecs, NVIDIA `akmod-nvidia`, libdvdcss); reviewed package maintainership, koji-style builds. | **AUR** — PKGBUILDs only (build recipes; no binary distribution, no signing); anyone can upload after creating an account; no vetting; deletion is post-hoc on report. **112,669 PKGBUILDs as of May 2026** (per the aur.archlinux.org/packages search interface, "112669 packages found. Page 1 of 2254"). Documented malware incidents: most notably July 2025, when user `danikpapas` uploaded three malicious packages — `librewolf-fix-bin`, `firefox-patch-bin`, and `zen-browser-patched-bin` — at ~20:00 UTC+2 on Jul 16, 2025, installing a Remote Access Trojan via scripts from a GitHub repo controlled by the attacker; packages were removed July 18 (BleepingComputer, Jul 22 2025). | COPR/RPM Fusion ship *signed binaries from reviewed maintainers in controlled infra*. AUR is *unreviewed source build scripts run with full user privileges at build time*. Arch's own docs and project leads explicitly say "AUR is random stuff from the internet; read every PKGBUILD." This is the single largest supply-chain delta. |
| **6. Packaging workflow for contributors** | RPM `.spec` file → `rpmbuild` locally → `fedpkg`/`mock` for clean chroot → submit via `fedora-review` (peer review) → import to dist-git → builds in **Koji** (centralized buildsystem) → push via **Bodhi** with karma gate. F44 switches dist-git CI from Zuul to **Packit**. | `PKGBUILD` (bash script with `pkgname`, `pkgver`, `source=`, `build()`, `package()`) → `makepkg` builds locally in a fakeroot; clean chroot via `extra-x86_64-build`/`devtools`. Official packages built by maintainers on their own systems and uploaded; Valve-funded centralized build infrastructure rollout since Sept 2024 (see row 27). AUR contributions skip all of this. | Fedora packaging is high-ceremony (review, koji, bodhi, FAS account, sponsor); Arch packaging is a one-file bash script. Trade-off: higher provenance + lower throughput vs. the opposite. |
| **7. Universal packaging & atomic variants** | Flatpak preinstalled; Flathub remote enabled by default in F44 Workstation; GNOME Software/Plasma Discover integrate it. **Snap not supported.** AppImage works but not integrated. Upstream atomic variants: **Silverblue** (GNOME), **Kinoite** (KDE), **Sericea** (Sway), **Onyx** (Budgie), **COSMIC Atomic**, **Sway Atomic**, **Budgie Atomic** — all use rpm-ostree/bootc with composefs. F44 ships sealed Atomic container images using systemd-boot + UKI. | Flatpak available (`pacman -S flatpak`) but **not preinstalled**; no Flathub remote enabled. AppImage runs (it's self-contained). **No upstream immutable/atomic variant.** Snap unsupported. | Fedora has a first-class story for both universal packaging and image-based OS; Arch users get classic-package Arch only — atomic Arch variants exist only as third-party derivatives (which are out of scope). |
| **8. Package count & lag from upstream** | ~82,150 binary RPMs across `fedora`+`updates` (May 2026). Lag varies: GCC/glibc/Mesa/GNOME usually within days of upstream tag during pre-freeze; security fixes typically <72 h via Bodhi. Major Python/Ruby/Go interpreter bumps are tied to release cycle. | ~14,000 binary packages in `core`+`extra`+`multilib`; AUR adds 112,669 PKGBUILDs. Lag for desktop and core libraries is often **same-day to a few days** from upstream tag. | Fedora has the larger curated set; Arch is leaner but supplements with AUR. On *individual* package freshness, Arch usually wins by days (especially for niche apps); on *coordinated rebases* (toolchain + DE), they're comparable. |
| **9. Parallel/multi-version packaging** | Versioned package names provide side-by-side installs (e.g., F44 has `mariadb10.11` and `mariadb11.8`, `python3.12`/`python3.13`/`python3.14`, `nodejs20`/`nodejs22`/`nodejs24`, `llvm21`-compat alongside LLVM 22, `gcc14`/`gcc15` compat libs). Modularity is largely retired but versioned RPMs remain. | Generally **one canonical version per package** in repos; legacy versions land in AUR (e.g., `python311`, `gcc11`). `linux` + `linux-lts` + `linux-hardened` + `linux-zen` are the main exception (parallel kernel installs supported by mkinitcpio presets). | If you need multiple SDK/runtime versions installed system-wide without containers, Fedora is materially easier. On Arch the idiomatic solution is `toolbox`/`distrobox`/Nix or AUR-built old versions. |

| **13. Installation experience** | **Anaconda** graphical installer on Live ISO (~2.5 GB). ### 5. System architecture and defaults

| Criterion | Fedora | Arch | Net difference / notes |
|---|---|---|---|
| **16. Init system & integration depth** | systemd 259.5; heavily integrated — preset files in `/usr/lib/systemd/system-preset/` enable many services on install; uses `systemd-resolved` by default, `systemd-oomd` enabled by default on Workstation, `systemd-homed` available, journal persistent by default, `systemd-tmpfiles` actively managed. Some Red Hat-led upstream features (sysext, mkosi-initrd, bootupd) shipped first. | systemd 259.x; near-vanilla configuration. **Presets are mostly "disable"** — installing a package does not enable its unit. `systemd-resolved` is not enabled by default; `/etc/resolv.conf` is plain. `systemd-oomd` not enabled. | Same software, opposite policy. Fedora treats systemd as an opinionated platform; Arch treats it as the binary you turn on. |
| **17. Services-on-install policy** | **Yes.** Installing `httpd`, `postgresql-server`, `sshd` etc. typically enables (sometimes starts) the unit due to RPM scriptlets and presets. Anaconda installs of Server enable `sshd` by default. | **No, explicitly.** Arch packaging guidelines forbid auto-enabling units. After `pacman -S nginx`, the service is installed but stopped/disabled until you `systemctl enable --now nginx`. | This is the most consequential single behavioral difference for sysadmins. On Arch, installation is inert; on Fedora, installation is activation. |
| **18. Default filesystem & snapshot story** | **Btrfs by default since F33**, enforced in F44 Workstation Live (Ext4 path not exposed in the streamlined flow). Two subvolumes (`@root`, `@home`); zstd level 1 compression by default. **F44 introduces `snapm` + `boom` + `dnf-plugin-snapm` integration**: on Btrfs installs Anaconda configures auto-snapshots before/after dnf transactions and registers GRUB2 boot entries for snapshots (rescue/rollback from boot menu). | **No default filesystem.** `pacstrap` doesn't format anything; the user runs `mkfs.*` manually. archinstall offers ext4 (recommended), btrfs, xfs, f2fs. No upstream snapshot tooling configured — snapper, btrfs-assistant, grub-btrfs are available but require manual setup. | Fedora ships a turnkey CoW + rollback story in F44; Arch ships the building blocks and a wiki page. If you want "install package → reboot to pre-install state from GRUB," Fedora 44 does that out of the box. |
| **19. Bootloader default** | **GRUB2 + shim** (Secure Boot signed) on all classic editions. F44 introduces **`bootupd`** as a unified bootloader updater (decoupling RPM transactions from EFI updates). **Atomic Desktops** in F44 use **systemd-boot + UKI** for sealed images. UKI support is shipped (`uki-direct`); Workstation can opt in via dracut. | User choice via archinstall: **systemd-boot is the default/recommended** (simplest, ships with systemd); GRUB, Limine, rEFInd also available. Manual install: anything; the wiki documents systemd-boot, GRUB, syslinux, efibootmgr, rEFInd. UKIs straightforward to build (mkinitcpio supports them natively). | Both support UKI; Fedora prefers GRUB+shim for Secure Boot ecosystem reasons, Arch defaults to systemd-boot for simplicity. Practically: Fedora is what you want on hardware with Secure Boot you can't disable; Arch is what you want if you want to throw out GRUB. |
| **20. Initramfs generator** | **dracut** (modular, systemd-aware, generates host-only or generic initramfs, easy to embed microcode and UKIs). F44 adds **`mkosi-initrd`** as an alternative, with prebuilt initrds delivered as RPMs planned for later releases. Kernel package updates auto-trigger dracut via `kernel-install`. | **mkinitcpio** by default (presets in `/etc/mkinitcpio.d/`, fallback image disabled by default in recent versions). Pacman hooks regenerate on `linux`/`linux-firmware` updates. **booster** and **dracut** available as alternatives in `extra`. | dracut is more "batteries included" and is what most enterprise tooling expects; mkinitcpio is more hackable and closer to the metal. Recovery story: dracut emergency shell vs mkinitcpio's `break=`. Practically interchangeable; the implication is mostly which docs you read. |
| **21. Audio stack defaults** | PipeWire **1.6.3** with WirePlumber as the session manager. JACK/PulseAudio compat shims preinstalled. F44 carries Red Hat's PipeWire performance/latency tweaks (notably for pro-audio and conferencing). | PipeWire **1.6.4** with WirePlumber, same architecture. No distro-specific tweaks; ships upstream defaults. | Identical at the architectural level; Fedora is slightly more configured, Arch is vanilla. |
| **22. Configuration philosophy** | Opinionated. `/etc/` is heavily pre-populated by RPM (cron.d, dnf, systemd presets, polkit rules, NetworkManager, firewalld zones, SELinux policy). Updates use `.rpmnew`/`.rpmsave` mechanism when local edits conflict. | Minimal. `/etc/` contains only what the package ships and what the user adds. Updates use `.pacnew`/`.pacsave`; pacman warns prominently after every transaction that produces them. Wiki convention is to use `/etc/<svc>/conf.d/*.conf` drop-ins rather than editing distro defaults. | Fedora's defaults are designed to be a working system; Arch's defaults are designed to be minimal. On Fedora you remove things you don't want, on Arch you add things you do. |

### 7. Security and supply chain

| Criterion | Fedora | Arch | Net difference / notes |
|---|---|---|---|
| **25. Security defaults** | **SELinux enforcing** with `targeted` policy by default on all classic editions. Secure Boot fully supported (signed shim + GRUB + kernel + modules; out-of-tree drivers must sign via MOK). LUKS2 disk encryption offered in Anaconda. System-wide **crypto-policies** (`update-crypto-policies` — DEFAULT/FUTURE/LEGACY/FIPS). FIPS mode via `fips-mode-setup` (transitioning to automation in F44). Hardening flags via `redhat-rpm-config`: `-fstack-protector-strong`, `-D_FORTIFY_SOURCE=3`, `-fcf-protection`, `-fstack-clash-protection`, full RELRO+BIND_NOW, PIE. F44 adds opt-in restriction of `ptrace`/`/proc` debug visibility for unprivileged users by default. | **No MAC by default.** AppArmor and SELinux kernel support is compiled in but neither LSM is enabled — `lsm=` not set. Secure Boot: not signed by default; `sbctl`-driven self-enrollment is the documented path. LUKS available in archinstall. **No system-wide crypto-policy framework.** Hardening flags in `/etc/makepkg.conf`: similar set (`-fstack-protector-strong`, `_FORTIFY_SOURCE=3`, full RELRO, PIE), but no FIPS path. | The largest security delta. Out of the box: Fedora confines services with SELinux, supports Secure Boot end-to-end, and offers FIPS; Arch ships with none of those enabled. Both have similar compiler hardening at the binary level. |
| **26. Security advisory & CVE response** | **Bodhi** issues `FEDORA-YYYY-xxxxxxxxxx` errata per update with CVE list, severity, karma; visible at `bodhi.fedoraproject.org`. SLA: critical security updates typically pushed to stable within 24–72 h. Dedicated Fedora Security Team. | **Arch Security Team** publishes `ASA-YYYYMM-N` advisories at `security.archlinux.org`; tracker exposes AVG (Arch Vulnerability Group) and CVE views. Two team-member ack required before publishing an ASA. SLA: Arch typically gets the *upstream fix* before there is news to advise on; ASA publication often lags the package update by hours-to-days. | Different philosophies: Fedora backports security fixes (and writes a formal advisory); Arch ships the upstream version that already contains the fix (and writes the advisory afterward, sometimes). End-state patch latency is similar; *advisory coverage* favors Fedora; *raw fix latency* favors Arch. |
| **27. Supply chain & build integrity** | All packages built in **Koji** (centralized, isolated, signed). Per-release signing keys (offline). **Reproducible builds: F44 formally adopts the "no fewer than 99% of package builds reproducible" target** (Fedora Project Wiki, "Changes/Package builds are expected to be reproducible," deferred from F43 and accepted for F44; Fedora Magazine F44 Beta post: "With this change, all package builds are expected to be reproducible in the F44 final release"), with a public `rebuilderd` instance and `fedora-repro-build` tool. Maintainer onboarding requires sponsorship + FAS account. | Official packages built by individual maintainers; **Valve-funded centralized build service and HSM-backed signing enclave announced 28 Sept 2024** by Arch Linux project lead Levente Polyak: "Valve is generously providing backing for two critical projects that will have a huge impact on our distribution: a build service infrastructure and a secure signing enclave. By supporting work on a freelance basis for these topics, Valve enables us to work on them without being limited solely by the free time of our volunteers" (rollout ongoing through 2026, per Phoronix). Per-maintainer signing keys; package DB itself signed. Reproducible-builds effort exists at `reproducible.archlinux.org`. Maintainer onboarding via Trusted User / Package Maintainer process. | Fedora's build infra has been centralized and isolated for ~two decades; Arch is in the middle of moving there. Reproducibility: both are credible, Fedora's stated target (≥99% in F44) is the most explicit number publicly. **AUR is the outlier — no build isolation, no signatures.** |

### 8. Performance and footprint

| Criterion | Fedora | Arch | Net difference / notes |
|---|---|---|---|
| **28. Footprint & out-of-the-box weight** | Workstation Live install lands at roughly 6–10 GB; idle RAM with GNOME 50 in the ~1.5–2 GB range based on community measurements (no formal Phoronix Fedora-44-vs-Arch-2026.05.01 boot-time benchmark exists). Boot includes SELinux relabel checks, dnf metadata refresh, gnome-software, abrt, tracker3. Workstation enables a fairly large default service set. | Minimal install (`base linux linux-firmware`) is around 800 MB and ~300–500 MB RAM idle pre-DE (community/wiki figures). With GNOME 50.1 installed, comparable to Fedora. | Like-for-like (same DE) the difference is small at runtime; the big delta is what *you started with*. Arch wins on minimal-footprint scenarios because Fedora always pulls in a curated default set. **No formal benchmark suite has been published for this specific pair; these are order-of-magnitude figures.** |
| **29. Build tuning** | x86_64 baseline is **still x86_64-v1** (Fedora has discussed v2 multiple times but not switched). **glibc-hwcaps** mechanism used to ship optimized library variants per microarch level (`/usr/lib64/glibc-hwcaps/x86-64-v{2,3,4}/`); F42+ extended this to executables via the `hwcaps-loader` Change. No upstream LTO/PGO mandate; many packages opt in. Multilib: yes, via `*.i686` RPMs in updates/fedora repos. | Baseline is **x86_64-v1**; RFC 0002 to ship x86_64-v3 builds was approved conceptually but **not deployed** as of May 2026. Per Arch RFC 0032 ("Arch Linux Ports"): "With RFC0002 we have tried to establish x86_64_v3 … our efforts for automation and build infrastructure were not yet sufficient … led to the effort stalling for a long period of time." Superseded by the RFC 0032 "Ports" framework, still in development. LTO/PGO: per-package, several core packages built with LTO. Multilib: yes, `[multilib]` repo. | Neither ships v3 binaries upstream in May 2026, despite both having mechanisms. If you want v3-optimized binaries, both upstreams force you to a downstream (ALHP for Arch, CentOS ISA SIG for the Red Hat side) — out of scope. |

### 9. Stability and update workflow

| Criterion | Fedora | Arch | Net difference / notes |
|---|---|---|---|
| **30. QA & update reliability** | Formal **blocker bug** process: alpha → beta → final go/no-go meetings; F44 was delayed twice (Apr 14 → 21 → 28) because of installer/KDE-setup blockers. **openQA** runs automated install/desktop tests. Bodhi karma gating per update. `updates-testing` exists for users who want pre-promotion access. | No release per se. New packages enter `*-testing`, sometimes get human review, then go to stable. | **31. ## Recommendations

**Choose Fedora 44 if:**
- You want SELinux enforcing, Secure Boot, FIPS, system-wide crypto-policies, and Bodhi errata without configuring any of them. *(Threshold to reconsider: you're willing to spend a day hardening Arch and forgo SELinux.)*
- You want Btrfs auto-snapshots with boot-from-snapshot in GRUB out of the box (new in F44).
- You want immutable/image-based desktop variants (Silverblue/Kinoite) or edge variants (IoT/CoreOS) from the upstream project.
- You need a non-x86_64 architecture (aarch64, ppc64le, s390x).
- Your career or codebase touches the RHEL family — Fedora is the upstream and skills transfer directly.

**Choose Arch if:**
- You want kernel 7.0 today and every other library at the freshest stable upstream version, with no curation layer between you and upstream.
- You will keep up with `archlinux.org/news` and `pacman -Syu` interactively.
- You want CUDA/NVIDIA/multilib/Steam from official signed repos without third-party channels.
- You want to learn Linux internals as a side effect of using the system, and treat ArchWiki as primary documentation.
- You want minimum opinionation: no MAC by default, no services on install, no preconfigured `/etc/`.

**Hybrid pattern that works:** Run Fedora Workstation as the host (snapshots, SELinux, Secure Boot) and use **Distrobox/Toolbox with an Arch container** for cutting-edge userland tools — both projects support this. The inverse (Arch host + Fedora toolbox) works for the same reason.

**Avoid Arch upstream for:** production servers (no LTS, no advisory push, no enterprise tooling), single-architecture deployments where x86_64 isn't an option, environments where unattended security patching with formal advisories is a compliance requirement.

**Avoid Fedora upstream for:** long-running production servers without a planned 6-monthly upgrade — CentOS Stream/RHEL exist downstream specifically for that and are out of scope here.

## Caveats

- **Kernel version reporting:** Fedora 44 GA kernel is **6.19.14-300.fc44**; news sources occasionally conflated Fedora's kernel with Arch's. Fedora's `updates` repo pushes new mainline series within a release (so 7.0.x is expected in F44 by mid-to-late May 2026, post-cutoff for this comparison).
- **Quantitative footprint numbers** (boot time, idle RAM, disk footprint) are *not* available as a formal published Phoronix Fedora-44-vs-Arch-2026.05.01 benchmark; numbers cited here are from community measurements and ArchWiki, and should be treated as orders-of-magnitude rather than precise data.
- **Arch package count (~14,000):** binary packages in `core+extra+multilib`. The AUR adds **112,669 PKGBUILDs** (May 2026) but those are recipes, not vetted binaries.
- **x86-64-v3 status:** Both projects have *proposed* v3 builds but neither *ships* them as the default in May 2026; this is one of the more commonly misreported facts. Per Arch RFC 0032, the v3 effort "led to the effort stalling for a long period of time."
- **Fedora KDE Plasma is now an Edition (since F42), not a Spin** — sometimes still listed as "Spin" in older docs.
- **Arch is x86_64-only upstream**; Arch Linux ARM is a separate, unofficial project explicitly disclaimed by Arch upstream.
- **AUR security**: the single biggest practical security trap. The July 2025 incident involved three malicious packages (`librewolf-fix-bin`, `firefox-patch-bin`, `zen-browser-patched-bin`) uploaded by user `danikpapas`, all installing a Remote Access Trojan via scripts from a controlled GitHub repository (BleepingComputer, Jul 22, 2025). The project's own position is that users are responsible for reading every PKGBUILD before building.
- **Mid-comparison data shifts:** Several rolling/updates-released versions cited (PipeWire 1.6.3 vs 1.6.4, Mesa 26.0.3 vs 26.0.6) are exactly the kind of difference that may invert by the time this is read — the structural conclusions hold regardless.
